01根因诊断（为什么不是泛泛加 reviewer）

**症状**：draft 每次出的 spec 都要手动送 codex review，每次都出 high。

**样本**：`kuril-ios/docs/mobile-dev-specs/base/region-switch-guide.spec.html` 改了 9 个 revision，其中 rev2/rev7/rev9 明确写「据 Codex review/二审」修。抽出数据岛逐条看，所有标注「Codex 指出/修正」的点：

- 误以为弹窗要和登录强切互斥 → 实际 `UserLoginManager+.swift:50` 强切是独立链路

- 误以为 `Kuril.firstLaunchStartTime` 能判新装 → 实际它只是时间戳、安装维度、卸载重置

- `showChangeRegionAlertIfNeeded` 在 `newRegion=nil` 时会弹空名弹窗 → 才补 AC-11 调用方 guard

- `currentRegionCode()` 未初始化返回空串、不回退 Locale → 才补 D-9

- 同步 getter 判不出网络失败 → 才改 AC-7 判定口径

- `updateAll` 异步延迟约 2s 的就绪竞态 → 才补 D-11/AC-12 整条会话内补弹

**根因（用户确认占绝大多数）**：没有一条是 spec 内部逻辑矛盾，全是 **spec 对「现有代码/系统真实行为」做了乐观假设，而假设错了**。draft 写「复用 X 现有能力」时把 X 的真实语义/边界/时序猜错，整个方案建在错误地基上。

**自举佐证**：本 spec 自己也反复犯同样的错——rev1 把 skill 路径写成 `skills/specforge-draft/`（实际 `skills/draft/`）、`verdictCounts` 说成在 nav（实际 masthead）；rev2 把自产边界锚到 `annotate-main.mjs:12`（实际判断在 :14）、打包清单锚 `bundle.mjs:8`（实际 :11）；rev3 把强制声明放 render gate（实际 render 单入口分不出新生成/存量，该落 check-ready）。每轮送 Codex review 当场被抓出。这恰是本需求要根治的「现状断言被猜错」——哪怕认真写也防不住，必须第二双眼睛去真代码核。

**两个直接含义**：①前移成纯文本 gate 拦不住——矛盾不在 spec 里，一致性检查看不到 firstLaunchStartTime 其实是安装维度；②泛泛读 spec 挑质量的 subagent 也拦不住——它读不到被改代码。唯一能抓到的 reviewer 必须能去真实 codebase 核验每条现状断言——这正是 codex 当时在干的事。

02新 draft 流程（插入点）

  1. 发散 → 拷问决策（不变）。
  2. 强制表态：spec 应显式声明顶层 assumptionReview:{applicable, reason}——这次 spec 到底涉不涉及对现有代码/系统的断言。applicable:false 也要写理由。此「强制」由 check-ready 交付关把守（D-15）：缺声明则交付被拒；render 不因缺声明拒绝，故不破坏无此字段的存量 spec（D-13/AC-11）。
  3. 结构化：applicable:true 时，写「复用 X 现有能力 / X 现在的行为是 Y」这类对现状的事实断言必记一条 assumptions[]，带 id(ASM-n，唯一) + claim + evidence(file:line)，verified 与 verifiedDigest 初值显式 null。claim/evidence/reason 不得空白。这是事中收紧——逼 draft 去读代码标证据而非猜。
  4. 渲染 spec.html：render 先做友好结构 gate（在 schema 校验之前、对错误类型防御不崩）——若带了 assumptionReview：①applicable:true 但 assumptions 空、②applicable:false 但 assumptions 非空、③任一 assumption 缺 evidence 或 claim/evidence 空白 → 拒绝并列问题 id。render **不强制** assumptionReview 存在（存量兼容），也不做交付 gate（按 effectiveVerified 三态全渲）。
  5. 条件判定：assumptionReview.applicable:false → 跳过下面 reviewer 步骤，直接进交付检查。
  6. 独立核验：applicable:true → 主 agent spawn 一个全新 context、有 codebase 访问的 Claude subagent，拿每条 evidence 去真代码核对 claim，逐条判 verified = true/false（不得 null）+ false 给非空 note。reviewer 输出 [{id, verified, note}]，id 集合须与 assumptions 完全相等。
  7. 回写重渲：draft 自带的 annotate-assumptions.mjs <spec.html> <results.json>，按 id patch 进 assumptions（禁改 claim/evidence）、给每条算 verifiedDigest=digest(claim,evidence)、true 时清旧 false note、整次原子写回（任一 id 未知/重复/缺失则整次拒绝）、原地重渲。复用 annotate 自产边界（非 specforge-draft 拒绝写回）。
  8. 交付 gate：独立 check-ready <spec.html>——复用 extractDataIsland 安全抽数据岛（不执行 HTML）→ schema 校验 → 判 readiness：①缺 assumptionReview → 拒交付（强制声明落点，D-15）；②applicable:true 且存在 effectiveVerified!==true（verified 非 true，或 verifiedDigest≠当前 digest 致失效→视为未核）→ 拒、列未过项；③applicable:false → 通过。修 claim/evidence 后 digest 自动失效需重核。
  9. 全 effectiveVerified===true（或 applicable:false）→ check-ready 通过、交付。代码后续演进致的陈旧不在本机制保障内（A-7 残余风险）。

细节（默认折叠，按需展开）

概述

给 specforge-draft 加一层「现状假设核验」，治根因：codex review 反复抓的 high 几乎全是 spec 对现有代码真实行为的乐观假设错了（如把安装维度的 firstLaunchStartTime 当成可判新装、把 newRegion=nil 时会弹空名的方法当成安全复用）。这类缺陷不在 spec 内部（一致性 gate 拦不到），在 spec 与 codebase 之间，纯读 spec 的 reviewer 也抓不到。双管齐下：①事中收紧——把散在 prose 里的现状断言提升成结构化 assumptions[]，每条强制带 evidence(file:line)，render 拒绝无证据/空白断言，逼 draft 去读代码而非猜；②事后独立核验——交付前 spawn 一个全新 context、有 codebase 访问的 Claude subagent，顺 evidence 逐条把 claim 拉到真代码核对，把 verified(true/false) 回写进 spec 重渲（带 ✓/✗ 徽标），check-ready 交付 gate 拦未核/未声明。经三轮 Codex review 收紧契约：强制 assumptionReview 声明（落 check-ready 交付关、render 不拦以兼容存量）堵漏记、独立 check-ready 解 gate 阶段边界、draft 自带 annotate-assumptions 保零依赖、verifiedDigest 指纹防 spec 自编辑致陈旧（代码演进陈旧为已知残余风险、本期不兜）、统一 effectiveVerified 派生消除渲染/gate 不一致。条件触发：applicable:false 的非代码/纯决策 spec 跳过 reviewer。reviewer 默认 Claude subagent 保零外部依赖，codex 可用时作异构模型可选增强。
